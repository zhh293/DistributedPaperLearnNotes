# 《TiDB: A Raft-based HTAP Database》通俗精讲

## 先说这篇论文到底在解决什么

这篇论文真正要解决的问题，不是“怎么再造一个分布式数据库”这么简单。

它要解决的是一个更具体、也更难的问题：

- 能不能在一个系统里，同时把事务处理和分析处理都做好？
- 而且不是“凑合能跑”，而是既要新鲜数据、又要性能隔离、还要高可用、还要强一致。

这就是 `HTAP`。

### 1. 什么是 HTAP

HTAP 是 `Hybrid Transactional and Analytical Processing`，也就是：

- 一边跑 OLTP 事务读写
- 一边跑 OLAP 分析查询

翻译成人话：

- 白天业务系统在疯狂写订单、改库存、更新账户
- 同时产品、运营、BI、风控还想立刻分析“最新数据”

而且他们不想等夜里跑 ETL，不想接受“分析看到的是几小时前的旧数据”。

### 2. 为什么这件事难

因为 OLTP 和 OLAP 的诉求几乎天然打架。

- OLTP 喜欢按行快速读写、低延迟、小事务
- OLAP 喜欢按列批量扫描、聚合、压缩、向量化执行

如果硬让它们共用一份数据、同一套资源，通常会出现两个问题：

1. `不新鲜`
   比如走 ETL，把事务库的数据定期同步到分析库，那分析看到的一定不是最新状态。
2. `互相拖累`
   比如事务和分析都打在同一份存储和计算资源上，大查询会直接把在线事务拖慢。

所以这篇论文真正要解决的核心矛盾是：

- 怎么既让分析看到足够新的数据，又不把事务处理拖垮？

### 3. 作者给出的答案是什么

作者的答案非常有意思：

- 不要把 `Raft` 只当成“高可用复制协议”
- 还可以把它扩展成“同时为事务副本和分析副本提供一致数据源”的基础设施

一句话总结：

- 这篇论文是在讲：如何把 `Raft` 的复制日志，变成一个 HTAP 系统同时满足“新鲜 + 隔离 + 一致 + 高可用”的底层桥梁。

---

## 先给你一个总览

你读完整篇论文，脑子里至少要留下下面 7 件事：

1. TiDB 把 HTAP 问题转化成“如何为不同工作负载维护一致但异构的副本”。
2. TiKV 用 `Raft` 管理行存副本，主要服务事务处理。
3. TiFlash 引入 `learner`，异步跟随 Raft 日志，把行存数据变成列存数据，主要服务分析处理。
4. `learner` 不参与选主和法定多数，因此不会像普通 follower 那样明显拖慢事务提交路径。
5. TiDB 的关键不是“多一份副本”，而是“多一份格式不同、用途不同、但保持一致的数据副本”。
6. 整个系统的关键工程挑战包括：multi-Raft 扩展、读写优化、Region 管理、日志回放、schema 同步、列存增量合并。
7. 这篇论文的真正价值，是把“共识算法的复制能力”扩展到了 HTAP 架构设计层面。

## 一、先把问题背景拎清楚

### 1. 旧世界为什么不行

过去常见做法通常是：

- OLTP 用事务数据库
- OLAP 用数仓 / Hadoop / 列存系统

中间再靠：

- ETL
- CDC
- 流式同步

把数据搬过去。

这个思路有个非常明显的问题：

- 分析看到的数据不够新

如果同步间隔是几小时，那就不是实时分析。
如果同步更频繁，复杂度和一致性问题又会越来越重。

### 2. 那能不能都放在一个系统里

也有人这么做：

- 让事务查询和分析查询直接读同一份数据

这又会撞上另一个问题：

- 大分析会影响在线事务

因为分析查询通常：

- 扫大量数据
- 吃很多 CPU
- 吃很多内存
- 访问模式和事务完全不同

所以，光“同库同表”不够，必须处理：

- 数据新鲜度
- 资源隔离
- 一致性

### 3. 为什么作者会想到 Raft

因为高可用分布式数据库本来就常常有副本。

而副本的维护，本来就靠共识协议。

作者意识到：

- 如果 `Raft` 能稳定维护一致副本
- 那为什么不顺着这条复制链，再挂出一类专门服务分析的副本？

这就是这篇论文最聪明的出发点：

- 不是额外搞一套旁路同步系统
- 而是直接沿着一致性复制主干，长出分析副本

## 二、TiDB 的核心想法到底是什么

### 1. 一句话版

TiDB 的核心想法可以压缩成一句话：

- 事务数据先按行存方式用 `Raft` 维护强一致副本，再把提交日志异步复制给 `learner`，由 learner 把它实时转成列存副本供分析查询使用。

### 2. 为什么不是简单多加几个 follower

这个问题很关键。

你可能会想：

- 既然要多一份副本，那多加几个 follower 不就好了？

问题在于，普通 follower 是 `Raft quorum` 的一部分。

这意味着：

- leader 复制时要考虑多数派
- follower 数量和行为会影响提交性能
- 它们也可能参与选主

如果你只是为了跑分析，直接把这种节点加进主提交路径，反而会拖慢 OLTP。

所以作者设计了 `learner`。

### 3. learner 到底是什么

论文里的 learner 可以理解成：

- “旁听副本”

它的特点是：

- 接收 leader 的日志复制
- 不参与 quorum
- 不参与 leader election
- 领导者给客户端响应时，不需要等 learner

这一下子就解决了核心矛盾：

- 可以得到一份相对新鲜的数据副本
- 但又不会显著增加事务提交路径上的同步负担

### 4. learner 上为什么要变成列存

因为 OLAP 查询很少是：

- 随机改一行

它更多是：

- 扫很多行
- 只用部分列
- 做聚合、join、过滤

这时候列存的优势非常明显：

- 更适合压缩
- 更适合向量化执行
- 更适合分析型扫描

所以 learner 不是简单“多拷一份行存”，而是：

- 把 Raft 日志重放后，物化成列存

这就是 TiFlash。

## 三、TiDB 的整体架构怎么理解

论文里的架构可以拆成三层：

1. `TiKV`
2. `TiFlash`
3. `SQL Engine + PD`

### 1. TiKV：行存事务层

TiKV 是：

- 行存
- 基于 key-value
- 每个 Region 一个 Raft group
- 负责事务读写、高可用、一致性

你可以把 TiKV 理解为：

- 真正承接 OLTP 事务数据的主战场

### 2. TiFlash：列存分析层

TiFlash 是：

- learner 节点构成的列存副本系统
- 从 TiKV 的 leader 异步拿日志
- 重放日志并转成列格式

它的目标是：

- 服务分析型查询
- 尽量和事务写路径隔离

### 3. PD：全局调度和时间戳中心

PD 负责：

- Region 管理
- 负载均衡
- 位置信息维护
- 全局单调递增时间戳

论文里还说：

- 这些时间戳也被用作事务 ID

这一步很重要，因为 TiDB 的事务层和数据分布管理层都依赖一个可靠的全局协调点。

### 4. SQL 层：同时会用行存和列存

TiDB 的 SQL 引擎不是盲目决定去哪读数据。

它会根据查询类型和代价，决定：

- 读 TiKV
- 读 TiFlash
- 或两边配合

所以 TiDB 的关键不是“我既有行存又有列存”。

真正关键是：

- 上层优化器知道两边都是什么，且两边数据又是强一致语义下的新鲜副本

## 四、multi-Raft 存储为什么是基础

### 1. 为什么要切 Region

数据一大，不可能整库只挂一个 Raft group。

所以 TiDB 把大 key-value 空间切成很多连续 key range：

- 每一段叫一个 `Region`

每个 Region：

- 有多副本
- 自己跑一套 Raft

这就是 `multi-Raft`

### 2. 这一步在解决什么问题

本质上是在解决：

- 单个 Raft 组无法扩展全库吞吐

把大表切成很多 Region 之后：

- 不同 Region 可以分布在不同机器
- 不同 leader 可以并行服务请求
- 整体读写吞吐就能扩起来

### 3. 但 multi-Raft 自己也会带来问题

Region 多了以后，会有一堆新麻烦：

- 元数据管理
- 心跳开销
- 热点 Region
- split / merge
- 跨节点负载均衡

所以 TiDB 不是“用了 Raft 就结束”，而是：

- 你把 Raft 用到全系统后，还得继续解决管理层复杂度

这也是论文很有工程味的地方。

## 五、TiKV 这部分最重要的优化是什么

论文里这部分特别值得细看，因为它说明：

- 标准 Raft 流程作为教材很清楚
- 但要把它做到生产可用，还得做大量工程优化

### 1. 先回顾标准 Raft 写路径

一个最朴素的流程是：

1. leader 收到请求
2. 本地追加日志
3. 发给 followers
4. 等多数派确认
5. 提交并 apply
6. 返回客户端

这个流程没错，但如果你完全串行执行，会比较慢。

### 2. TiKV 的优化思路

TiKV 做了几类关键优化：

- 本地 append 和发给 follower 并行化
- follower 复制做批量发送
- leader 不必每一步都傻等，可以乐观推进再修正
- apply 阶段异步交给其他线程

翻译成人话就是：

- 只要不破坏一致性，就尽量把“必须串行”的步骤变少，把“可以并行”的部分榨出来

### 3. 为什么这类优化重要

因为大规模场景里，Raft 的瓶颈往往不在“理解它的安全性证明”，而在：

- 网络往返
- 磁盘刷写
- 单线程串行处理

而这些恰好是工程优化最能起作用的地方。

## 六、读请求为什么还要专门优化

很多人刚学共识协议时会有一种误解：

- 读不就是比写简单吗？

但在强一致系统里，读也不简单。

### 1. 为什么 leader 读不能随便直接回

因为你得确认：

- 它现在真的是 leader
- 它的状态足够新

否则网络分区或主从切换后，就可能读到过时数据。

### 2. 论文里的两个关键优化

#### `read index`

leader 记录一个当前提交位置，然后通过心跳确认自己还是合法 leader；
只有本地 apply 到足够新位置后，才返回读结果。

#### `lease read`

在一个租约期间内，leader 被认为不会被替换。
于是它可以直接在本地处理读请求，而不必每次都发网络确认。

### 3. follower read 又在解决什么

如果所有读都打 leader，热点 Region 会把 leader 压爆。

所以论文还支持 `follower read`：

- follower 先拿最新 read index
- 确认自己本地状态够新
- 再返回客户端

这一步本质上是在做：

- 强一致前提下的读流量分摊

这个点很重要，因为它说明：

- 高一致性系统不是只能“所有流量都打 leader”

## 七、Region 管理为什么是大规模系统里绕不过去的问题

### 1. 为什么要 split

如果某个 Region：

- 太大
- 太热

那它就会成为瓶颈。

所以必须拆。

### 2. 为什么要 merge

反过来，如果 Region 太碎：

- 元数据太多
- 心跳太多
- 管理开销太大

那又会浪费 CPU 和网络。

所以必须合。

### 3. split 为什么能比较轻

论文强调，split 本身主要是：

- 元数据变化

它不要求立刻重写大量真实数据。

所以成本相对较低。

### 4. merge 为什么更复杂

merge 牵涉两个 Raft group 的协调。

这不像 split 那么自然。

所以它需要：

- 更复杂的阶段化处理
- 先停止一部分服务，再并入另一部分

你在这里能看到一个很真实的系统规律：

- “把东西拆开”常常比“把东西重新合回去”容易

## 八、TiFlash 真正的难点是什么

很多人看到 TiFlash 会觉得：

- 哦，不就是把日志抄一份到列存吗

其实远没有这么简单。

### 1. 第一难点：日志不是直接等于列数据

事务日志里有很多中间状态：

- prewrite
- commit
- rollback

TiFlash 不能傻乎乎把所有日志都照单全收。

它必须先做：

- 无效日志压缩
- 事务状态整理
- 把最终有效变更解码成真正的数据变更

也就是论文里说的：

- compact logs
- decode tuples
- transform to columnar

### 2. 第二难点：schema 可能在变

TiKV 那边的数据本质上是字节编码。

但 TiFlash 想把它实时变成列存，就必须知道当前 schema。

如果 schema cache 过期了，就会出问题。

所以论文设计了两段式 schema 同步：

- 常规同步
- 遇到不匹配时强制同步

这一步其实很有代表性：

- 任何“异步物化副本”系统，只要目标结构是 schema-aware 的，就一定绕不开 schema 同步问题

### 3. 第三难点：列存也要处理高频更新

很多人会下意识觉得：

- 列存天然适合分析

没错，但它通常不天然适合高频小更新。

所以 TiFlash 设计了 `DeltaTree`：

- 新来的增量先放 delta space
- 稳定数据放 stable space
- 后续再 merge

这其实和 LSM 思想有点像：

- 先便宜地接住写入
- 再后台整理成更适合读的形态

这个设计的意义就在于：

- 列存不是静态仓库
- 在 HTAP 里它必须“边接近实时更新、边保持分析友好”

## 九、TiDB 这篇论文最核心的机制，我替你压缩成 5 条

### 1. 用 `Raft` 维护事务行存副本

保证：

- 一致性
- 高可用
- 可扩展

### 2. 引入 `learner`

保证：

- 分析副本可以跟上日志
- 但不进入主提交法定多数

### 3. 在 learner 上把行存转列存

保证：

- OLAP 查询有更合适的数据组织方式

### 4. 把 OLTP 和 OLAP 放到不同资源上

保证：

- 尽量隔离性能干扰

### 5. 用统一优化器选择访问路径

保证：

- 查询能按代价决定走行存、列存或混合路径

## 十、这篇论文真正聪明的地方在哪

### 1. 它不是把 OLAP 当成“事后同步产品”

而是把 OLAP 副本纳入一致性复制体系的一部分。

这点非常关键。

很多系统里分析副本总有一种“旁路、松耦合、尽力而为”的味道。

TiDB 这里的思路更强：

- 虽然 learner 不在主 quorum 里
- 但它的数据来源仍然是同一条复制主干

### 2. 它抓住了 HTAP 的本质不是“同一份数据”，而是“同一份事实的不同物理表示”

这是这篇论文最值得学习的抽象。

HTAP 的难点不是：

- “让 OLTP 和 OLAP 共享同一份存储文件”

而是：

- “让它们基于一致的新鲜事实，各自拥有适合自己工作负载的物理组织方式”

### 3. 它没有神化 Raft，而是工程化地改造它

这篇论文没有停留在：

- “我们用了 Raft，所以系统就强了”

它认真去解决：

- 提交路径怎么并行
- 读路径怎么降网络开销
- 多 Region 怎么调度
- learner 怎么低延迟回放
- schema 怎么同步

这说明：

- 共识协议只是基础，系统最终成不成，还得看上层工程能力

## 十一、这篇论文的代价和边界也要看清

### 1. learner 是异步的，所以分析副本不是零延迟

它强调的是：

- 新鲜

但不是：

- 与事务提交完全零延迟同步

所以这里要理解成：

- “足够新、低延迟、强一致视图可控”

而不是神话成“分析副本和事务副本瞬间完全同步无差”

### 2. 系统复杂度明显提高

你不再只维护：

- 行存 Raft

你还要维护：

- learner
- 列存物化
- schema 同步
- DeltaTree
- 混合查询优化

所以它不是一个“白捡收益”的系统。

### 3. 隔离不是绝对零影响

虽然 OLTP 和 OLAP 分资源部署，但：

- 日志复制
- 后台同步
- 元数据协调

仍然会有一定共享开销。

只是相比“事务分析共用一份主副本资源”，干扰已经小得多。

- Raft 不只是“选主 + 复制日志 + 提交”

你平时学 Raft 可能更关注：

- 安全性

这会把你对 Raft 的理解，从“算法”提升到“系统骨架”层面。

## 十三、Q&A：把最容易冒出来的问题堵住

### Q1：TiDB 为什么不直接让 TiKV 同时服务 OLTP 和 OLAP？

因为行存和事务路径更适合 OLTP，大量分析查询会带来明显资源争用和性能干扰。

### Q2：为什么 learner 不参与 quorum 这么重要？

因为这样它就不会拖慢主写路径。leader 给客户端成功响应时，不需要等待 learner。

### Q3：既然 learner 异步复制，那怎么谈一致性？

事务提交的一致性仍由 TiKV 的 Raft 保证；分析读取的一致性则通过读时约束和日志顺序重放来保证一个正确、可控的新鲜视图。

### Q4：为什么分析副本一定要变成列存？

因为分析查询通常按列扫描、聚合、过滤，列存更省 IO、更易压缩、更适合向量化处理。

### Q5：TiFlash 最难的工程点是什么？

不是“存成列”本身，而是如何把事务日志可靠、低延迟、按顺序地转成列存，同时处理 schema 变化。

### Q6：multi-Raft 为什么不是简单并行化就完了？

因为 Region 数量一多，就会带来热点、元数据、心跳、split/merge、资源均衡等大量系统管理问题。

### Q7：TiDB 这篇论文最值得学的是什么？

是这种思维方式：把共识协议的复制能力，从“高可用副本同步”扩展成“异构副本协同”的系统架构能力。

### Q8：这篇论文是不是说明所有 NewSQL 都应该照抄这个设计？

不是。它说明这是一个很强的方向，但具体是否适合，还取决于 workload、资源成本、复制延迟容忍度和系统复杂度预算。

## 十四、最后给学生的一段收束

如果你以前觉得：

- 共识算法负责一致性
- 分析系统负责分析
- 事务系统负责事务

那这篇论文会把这些边界重新揉在一起。

它告诉你：

- 共识协议不仅能保副本一致
- 还能成为异构存储副本的同步主干
- 再配合合适的物理存储格式和优化器，就能做出真正实用的 HTAP 系统

所以这篇论文真正厉害的地方，不在于它又发明了一个新共识算法。

而在于它做了一件更成熟的事：

- 把一个已有的强一致复制机制，扩展成了一整套服务于 HTAP 的系统架构。

如果你把这件事看懂了，你对 Raft 的理解就已经不只是“论文知识点”，而是开始进入“它在真实工业系统里到底能长成什么样”的阶段了。
