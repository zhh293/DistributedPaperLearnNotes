# 《The Log-Structured Merge-Tree》深度讲解

## 零、先从一个你一定能感受到的痛点说起

假设你在运维一个 MySQL 数据库，有一张"操作日志表"——每秒写入 5000 条记录，每条记录带有 user_id 字段，需要按 user_id 建索引。

你会发现一个问题：数据库的写入 TPS 做不上去。`iostat` 一看，磁盘的 `%util` 已经打满，但吞吐量（MB/s）其实很低。为什么？

答案是：**MySQL（InnoDB）底层用的是 B+ 树索引。每来一条新记录，它都要找到 B+ 树中 user_id 对应的叶子页，把那一页从磁盘读出来，插入新条目，再写回去。** 当 user_id 的分布是随机的时候，这些叶子页散布在磁盘各处——这就是大量的随机 I/O。

机械磁盘的随机 I/O 大约每秒只能做 100~200 次（受磁头寻道时间限制），即使 SSD 也只有几万到十几万 IOPS。而顺序写入，机械磁盘可以做到 100+ MB/s，SSD 可以做到 GB/s 级别。

**这就是 LSM-Tree 要解决的核心问题：如何把"高频随机写索引"这件事，改造成"低频顺序写"，从而释放存储介质的真正带宽。**

## 一、先把 B+ 树的瓶颈说透

### 1.1 B+ 树的结构长什么样

```
                    [根节点: 50|100]
                   /        |        \
          [20|35]        [70|85]      [120|150]
         /   |   \      /   |   \     /   |   \
      [叶子页]  [叶子页]  ...  [叶子页]  ...  [叶子页]
       ↕ 双向链表连接 ↕
```

B+ 树的关键特性：

- 所有实际数据/索引条目都存在**叶子节点**（叶子页）中
- 叶子节点之间通过双向链表连接，支持范围查询
- 内部节点只存"路由键"，用来指引搜索方向
- 树高一般 3~4 层（对于百万级到亿级数据）
- 每个节点对应磁盘上的一个**页（page）**，通常 16KB（InnoDB 默认）

### 1.2 B+ 树为什么读很快

查找一个 key 的过程：从根页出发，每层做一次二分比较确定走哪个子指针，3~4 次页访问就能到叶子。对于热数据，上层内部节点基本常驻 Buffer Pool（内存缓存），实际磁盘 I/O 可能只有 1 次（读叶子页）。

这就是 B+ 树的核心优势：**点查和范围查询都很高效，因为数据本身就组织在"最终位置"。**

### 1.3 B+ 树写入的真正瓶颈在哪

现在来看写入。当你 INSERT 一条记录时，InnoDB 需要：

1. 从根页出发，定位到目标叶子页
2. 如果该叶子页不在 Buffer Pool 中 → **随机读** 一次磁盘
3. 在内存中修改该页（插入新条目）
4. 标记脏页，等待 checkpoint 时 **随机写** 回磁盘

问题核心：**当写入的 key 是随机分布的（比如 UUID、随机 user_id），每次写入大概率打到不同的叶子页。** 如果数据量超过 Buffer Pool 容量，大部分叶子页都是冷的，每次写入都意味着一次随机读 + 一次随机写。

具体算一笔账：

- 假设表有 1 亿条记录，索引叶子页约 60 万页（每页 16KB，装约 170 条索引）
- Buffer Pool 8GB，能缓存约 50 万页
- 当索引大小超过 Buffer Pool，写入就开始频繁 miss

这时候：

- 每次插入 ≈ 1 次随机读（把目标叶子页调入内存）+ 1 次随机写（脏页刷盘）
- 机械磁盘随机 IOPS ≈ 150 → 写入 TPS 天花板 ≈ 150
- SSD 随机 IOPS ≈ 50000 → 写入 TPS 天花板 ≈ 50000

但如果同样的数据量用顺序写：

- 机械磁盘顺序写 ≈ 150 MB/s → 每秒可写约 1000 万条索引记录
- SSD 顺序写 ≈ 2 GB/s → 更加夸张

**随机写和顺序写之间存在 100~1000 倍的性能鸿沟。这就是 LSM-Tree 要利用的核心差距。**

### 1.4 B+ 树的"写放大"问题

还有一个隐蔽但严重的问题：**写放大（Write Amplification）**。

在 B+ 树中：

- 你只想写入一条 100 字节的索引记录
- 但 InnoDB 必须把整个 16KB 的页读出来、修改、再写回去
- 实际写入磁盘的数据量是逻辑修改量的 160 倍

更极端的情况——页分裂：

- 当一个叶子页满了，B+ 树要把它拆成两页
- 这涉及新页的分配、父节点的更新、可能的级联分裂
- 一次逻辑写入可能引发多个页的物理写入

**总结 B+ 树的写入瓶颈：**
1. **随机 I/O**：key 分布随机 → 打到随机叶子页 → 磁盘寻道时间主导
2. **写放大**：修改 100 字节 → 读写 16KB 页 → 比实际数据大 100+ 倍
3. **页分裂开销**：在写入热路径上可能触发复杂的结构调整

## 二、LSM-Tree 的底层结构到底长什么样

### 2.1 宏观架构

```
┌─────────────────────────────────────────────────────────┐
│                     写入请求                              │
└───────────────────────────┬─────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────┐
│  WAL (Write-Ahead Log)  ← 先写日志，保证持久性             │
│  顺序追加写，非常快                                        │
└───────────────────────────┬─────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────┐
│  MemTable (C0)  ← 内存中的有序数据结构                     │
│  通常是跳表(SkipList)或红黑树                              │
│  大小：64MB ~ 256MB                                      │
│  写入：O(log N) 内存操作，零磁盘 I/O                       │
└───────────────────────────┬─────────────────────────────┘
                            │ MemTable 写满 → 冻结为 Immutable MemTable
                            │ → Flush 到磁盘成为 SSTable
                            ▼
┌─────────────────────────────────────────────────────────┐
│  Level 0 (L0)  ← 刚从内存刷下来的 SSTable 文件             │
│  文件之间 key 范围可能重叠                                  │
│  每个文件内部有序                                          │
└───────────────────────────┬─────────────────────────────┘
                            │ Compaction（归并压缩）
                            ▼
┌─────────────────────────────────────────────────────────┐
│  Level 1 (L1)  ← 文件之间 key 范围不重叠                  │
│  总大小 = L0 大小 × 放大因子(通常 10x)                     │
└───────────────────────────┬─────────────────────────────┘
                            │ Compaction
                            ▼
┌─────────────────────────────────────────────────────────┐
│  Level 2 (L2)  ← 更大的层                                │
│  总大小 = L1 大小 × 10                                    │
└───────────────────────────┬─────────────────────────────┘
                            │
                            ▼
                          ......
                            │
                            ▼
┌─────────────────────────────────────────────────────────┐
│  Level N (最底层)  ← 存放最老/最大的数据                    │
│  总大小 = L(N-1) × 10                                    │
└─────────────────────────────────────────────────────────┘
```

### 2.2 SSTable：LSM-Tree 的磁盘存储单元

SSTable（Sorted String Table）是 LSM-Tree 在磁盘上的基本存储单元。每个 SSTable 文件的内部结构：

```
┌──────────────────────────────────────────┐
│           Data Block 1                    │  ← 按 key 排序的 KV 对
│   key1:value1, key2:value2, ...          │     通常 4KB 一块
├──────────────────────────────────────────┤
│           Data Block 2                    │
├──────────────────────────────────────────┤
│           ...                             │
├──────────────────────────────────────────┤
│           Data Block N                    │
├──────────────────────────────────────────┤
│         Meta Block (Bloom Filter)         │  ← 布隆过滤器：快速判断 key 不存在
├──────────────────────────────────────────┤
│         Meta Block (Stats)                │  ← 统计信息
├──────────────────────────────────────────┤
│          Index Block                      │  ← 每个 Data Block 的最大 key + offset
├──────────────────────────────────────────┤
│           Footer                          │  ← 元信息入口
└──────────────────────────────────────────┘
```

关键特性：

- **文件一旦写入就不可变（Immutable）**：只追加，不修改
- **内部有序**：key 按字典序排列，支持二分查找
- **自带索引**：Index Block 记录每个 Data Block 的边界 key，查找时先定位 Block 再块内搜索
- **自带布隆过滤器**：一次内存探测即可判断 key 大概率不存在，避免无效磁盘读

### 2.3 MemTable：内存中的写入缓冲区

MemTable 是一个内存中的有序数据结构，在 RocksDB 中默认实现为**跳表（SkipList）**。

为什么选跳表而不是红黑树？

- 跳表支持无锁并发读、细粒度锁并发写（CAS 操作）
- 红黑树的旋转操作在并发场景下需要复杂的加锁
- 跳表的内存局部性在实际测试中也不差

MemTable 的生命周期：

```
Active MemTable (可读可写)
    │  写满（达到 write_buffer_size，默认 64MB）
    ▼
Immutable MemTable (只可读，后台线程正在刷盘)
    │  Flush 完成
    ▼
L0 SSTable 文件 (磁盘上)
```

RocksDB 允许同时存在多个 Immutable MemTable（`max_write_buffer_number`），避免刷盘速度跟不上写入速度时阻塞前台写入。

## 三、写入路径：为什么 LSM-Tree 写入快

### 3.1 一次写入的完整路径

当应用调用 `db->Put(key, value)` 时：

```
1. 将 (key, value, seq_num, type) 追加写入 WAL 文件  ← 顺序写，极快
2. 将条目插入当前 Active MemTable                     ← 纯内存操作
3. 返回成功                                           ← 完成！
```

就这么简单。对比 B+ 树的写入路径：

```
1. 写 redo log（WAL）                                ← 顺序写
2. 在 Buffer Pool 中定位目标叶子页                    ← 可能触发随机读
3. 修改页内容                                        ← 内存操作
4. 标记脏页                                          ← 后续需要随机写回
```

LSM-Tree 写入之所以快，核心在于：**前台写入路径上完全没有随机 I/O。WAL 是顺序追加，MemTable 是纯内存操作。**

### 3.2 那数据最终怎么落到磁盘

数据从内存到磁盘的过程分两步：

**第一步：Flush（内存 → L0）**

当 MemTable 写满后，后台线程将其序列化为一个 SSTable 文件写入 L0。这个过程是：

- 遍历 MemTable（因为 SkipList 有序，遍历就是按 key 顺序输出）
- 按顺序写入 SSTable 文件的 Data Block
- 构建 Index Block 和 Bloom Filter
- 一次顺序写入完成

**第二步：Compaction（L0 → L1 → L2 → ...）**

后台线程定期将上层的 SSTable 文件与下层归并，产生新的、更大的、key 范围不重叠的 SSTable 文件。

```
Compaction 过程（以 L0 → L1 为例）：

L0:  [a-m] [d-z] [b-k]  ← 三个文件，key 范围有重叠
                 │
                 │  选取 L0 文件 + L1 中 key 范围重叠的文件
                 │  做多路归并排序
                 ▼
L1:  [a-f] [g-m] [n-s] [t-z]  ← 归并后的结果，key 范围互不重叠
```

Compaction 的本质：**多路归并排序。** 输入是多个有序文件，输出是一批新的有序文件。全程是顺序读 + 顺序写，没有随机 I/O。

### 3.3 写入为什么快的数学直觉

假设系统每秒写入 R 条记录，每条 100 字节，MemTable 大小为 64MB：

- MemTable 可以缓冲约 64万条记录
- 大约每 128 秒 Flush 一次
- Flush 时顺序写 64MB 数据，机械磁盘耗时 ≈ 0.5 秒

对比 B+ 树：

- 每条记录 1 次随机 I/O ≈ 7ms（机械磁盘）
- 每秒 5000 条 → 需要 35 秒的随机 I/O 时间 → 根本做不到

**LSM-Tree 通过将 64 万次随机写合并为 1 次 64MB 顺序写，实现了约 1000 倍的 I/O 效率提升。**

## 四、读取路径：LSM-Tree 为什么读取更复杂

### 4.1 点查（Get）的完整路径

查找一个 key 时，需要从新到旧逐层搜索：

```
1. 查 Active MemTable          ← 内存，O(log N)
2. 查 Immutable MemTable(s)    ← 内存，O(log N)
3. 查 L0 的每个 SSTable        ← 可能需要查多个文件（L0 有重叠）
4. 查 L1 的对应 SSTable        ← 二分定位到唯一一个文件
5. 查 L2 的对应 SSTable        ← 同上
6. ...直到找到或确认不存在
```

每一层的查找过程：

```
SSTable 查找：
1. 检查 Bloom Filter → 如果返回"不存在"，跳过此文件（避免 99%+ 的无效读）
2. 二分搜索 Index Block → 定位到目标 Data Block
3. 读取 Data Block → 块内二分搜索找到 key
```

### 4.2 为什么读取没有 B+ 树那么快

- B+ 树点查：1~2 次磁盘 I/O（大部分情况只需读 1 个叶子页）
- LSM-Tree 点查（最坏情况）：每层各读一个 Data Block，L0 可能要读多个文件

但实际工程中，多种优化让 LSM 读取没有理论最坏情况那么糟：

1. **Bloom Filter**：假阳率 1%，意味着 99% 的 "不在此文件" 判断无需磁盘 I/O
2. **Block Cache**：热点 Data Block 缓存在内存中
3. **索引缓存**：SSTable 的 Index Block 和 Filter Block 通常常驻内存
4. **时间局部性**：最近写入的数据大概率还在 MemTable 或 L0 中，直接内存命中

### 4.3 范围查询（Scan）

范围查询需要对所有层做归并迭代器：

```
创建一个归并迭代器（MergeIterator）：
  - MemTable 的迭代器
  - 每个 L0 SSTable 的迭代器
  - L1 的迭代器（因为不重叠，可以用 Concatenating Iterator）
  - L2 的迭代器
  - ...

按 key 顺序输出，遇到同一个 key 的多个版本取最新的
```

范围查询是 LSM-Tree 相对 B+ 树劣势最明显的地方——B+ 树叶子节点通过链表直接串联，扫描开销极低。

## 五、Compaction：LSM-Tree 的心脏

### 5.1 为什么必须做 Compaction

如果只有 Flush 没有 Compaction：

- L0 文件越来越多，key 范围严重重叠
- 读取时需要查越来越多的文件 → 读性能持续恶化
- 已删除/被覆盖的旧数据永远不会被清理 → 空间无限膨胀

Compaction 的作用：

1. **减少文件数和层数重叠** → 改善读性能
2. **清理过期数据（被覆盖的旧版本、已删除的 tombstone）** → 回收空间
3. **重新组织数据** → 保持每一层 key 范围不重叠的不变量

### 5.2 两种主流 Compaction 策略

**策略一：Leveled Compaction（RocksDB 默认）**

```
规则：
- L0 文件数超过阈值 → 选取 L0 文件 + L1 重叠文件 → 归并写入 L1
- Ln 总大小超过阈值 → 选取 Ln 中一个文件 + L(n+1) 重叠文件 → 归并写入 L(n+1)
- 每层（L1 起）内部文件 key 范围不重叠

特点：
- 读放大小（每层最多查 1 个文件）
- 空间放大小（每个 key 最多存在于相邻两层）
- 写放大大（一条数据从 L0 到最底层，每层都要被归并一次）
```

写放大的计算：假设放大因子为 10，层数为 7

- 一条数据从 MemTable flush 到 L0：写 1 次
- L0 → L1 compaction：写 1 次
- L1 → L2 compaction：最坏情况下被重写 10 次（因为 L2 是 L1 的 10 倍大）
- ...
- 总写放大 ≈ 10 × (层数 - 1) ≈ 50~60 倍

**策略二：Size-Tiered Compaction（Cassandra 默认、RocksDB 的 Universal Compaction）**

```
规则：
- 同一层积累到 N 个大小相近的文件 → 全部归并成一个更大的文件放到下一层

特点：
- 写放大小（每层只归并一次）
- 空间放大大（同一个 key 可能在多层都存在未被清理的旧版本）
- 读放大大（同一层内文件 key 范围重叠，读取需查多个文件）
```

### 5.3 Compaction 的工程影响

Compaction 是 LSM 系统最大的工程挑战：

- **占用 I/O 带宽**：大规模 compaction 时会抢占前台读写的磁盘带宽
- **占用 CPU**：归并排序、压缩/解压缩、计算布隆过滤器
- **写入停顿（Write Stall）**：如果 compaction 跟不上写入速度，L0 文件堆积超限，前台写入被阻塞
- **后台调度复杂性**：如何在不影响前台延迟的前提下最大化 compaction 吞吐

## 六、删除和更新的处理方式

### 6.1 删除：Tombstone 机制

LSM-Tree 不会去磁盘上找到旧数据物理删除它（因为 SSTable 不可变），而是写入一个特殊的**删除标记（Tombstone）**：

```
Put(key="user_123", value="Zhang San")   → 写入 MemTable
...
Delete(key="user_123")                    → 写入一个 Tombstone: (key="user_123", type=DELETE)
```

在读取时：

- 如果先遇到 Tombstone，就知道这个 key 已被删除，不再往下层查

在 Compaction 时：

- Tombstone 和它覆盖的旧记录相遇 → 两者都被丢弃 → 空间真正回收

**Tombstone 的生命周期**：只有当 Tombstone 被 compact 到足够底层、确保不存在更旧的记录后，才能被安全丢弃。

### 6.2 更新：覆盖写

更新一个 key 的值不需要找到旧值：

```
Put(key="user_123", value="Zhang San", seq=100)   → 第一次写入
Put(key="user_123", value="Li Si", seq=200)       → 更新，直接写入新版本
```

读取时从新到旧搜索，先找到 seq=200 的版本就返回，seq=100 的旧版本在后续 Compaction 中被清理。

## 七、RocksDB 中 LSM-Tree 的完整实战运用

### 7.1 RocksDB 是什么

RocksDB 是 Facebook 基于 Google LevelDB 开发的高性能嵌入式 KV 存储引擎。它是 LSM-Tree 在工业界最成功的实现，被广泛用作其他数据库的底层引擎：

- **TiKV（TiDB 的存储层）**：分布式事务 KV，底层就是 RocksDB
- **CockroachDB**：分布式 SQL 数据库，存储层基于 Pebble（Go 版 RocksDB）
- **MyRocks（MySQL + RocksDB）**：Facebook 自研的 MySQL 存储引擎替代品
- **Flink RocksDB State Backend**：流计算框架的状态存储
- **Kafka Streams**：流处理的本地状态存储

### 7.2 RocksDB 的架构全景

```
┌────────────────────────────────────────────────────────────────┐
│                        应用层 API                               │
│   Put / Get / Delete / Merge / Iterator / WriteBatch           │
└───────────────────────────────┬────────────────────────────────┘
                                │
┌───────────────────────────────┼────────────────────────────────┐
│                         Write Path                              │
│  ┌─────────────┐    ┌────────────────┐    ┌──────────────┐    │
│  │ Write Ahead │    │ Active MemTable│    │  Immutable   │    │
│  │    Log      │    │  (SkipList)    │    │  MemTable(s) │    │
│  └─────────────┘    └────────────────┘    └──────────────┘    │
│                                                                │
├────────────────────────────────────────────────────────────────┤
│                         Read Path                               │
```

| 优化点 | 原始论文 | RocksDB 实现 |
|--------|---------|-------------|
| MemTable 结构 | 泛指 AVL/B-Tree | SkipList（支持无锁并发读）|
| Compaction 策略 | Rolling merge | Leveled / Universal / FIFO 可选 |
| 读优化 | 无 | Bloom Filter + Block Cache + 前缀索引 |
| 删除机制 | 延迟标记 | Tombstone + TTL + CompactionFilter |
| 并发控制 | 概念性描述 | WriteBatch 原子写 + MVCC snapshot |
| 压缩 | 无 | Snappy / LZ4 / Zstd 逐层压缩 |
| WAL | 依赖外部事务日志 | 内置 WAL + 可配置 sync 策略 |

## 八、LSM-Tree 的核心代价——读放大、写放大、空间放大

任何工程决策都有 trade-off，LSM-Tree 用"延迟整理"换来了写吞吐，但代价体现在三个维度：

### 8.1 写放大（Write Amplification）

一条数据从写入到最终稳定，可能经历多次 Compaction 搬运。在 Leveled Compaction 中，数据从 L0 逐层合并到 Lmax，每过一层就被重写一次。如果层间大小比为 10，那理论上一条数据最终被写入磁盘的总次数约为层数×10 的量级。

这是 LSM 最被诟病的问题之一：虽然前台写很快，但后台总写入量远大于实际数据量，对 SSD 寿命（写入磨损）影响尤其大。

### 8.2 读放大（Read Amplification）

点查一个 key 时，最坏情况要从 MemTable 查到最底层。虽然 Bloom Filter 能跳过大部分不包含目标 key 的 SSTable，但范围查询（Range Scan）时每层都可能贡献数据，无法跳过。

### 8.3 空间放大（Space Amplification）

同一个 key 可能同时存在于多层（旧版本尚未被合并清理），加上 tombstone 标记占用空间，实际磁盘占用可能是有效数据的 1.1 ～ 2 倍。

### 8.4 三者不可兼得

这三个放大因子之间存在此消彼长的关系：

- Leveled Compaction：空间放大低、读放大低，但写放大高
- Size-Tiered / Universal Compaction：写放大低，但空间放大和读放大可能更高
- 没有一种策略能同时最小化三者，只能根据业务场景选取平衡点

## 九、B+ 树的瓶颈到底在哪——深入对比

### 9.1 B+ 树的结构回顾

MySQL InnoDB 的底层索引是 B+ 树。B+ 树的特点是：所有数据存储在叶子节点，叶子节点之间通过双向链表相连，内部节点只存索引键值用于路由。每个节点对应磁盘上一个 page（通常 16KB）。

### 9.2 B+ 树在写密集场景的具体瓶颈

**瓶颈一：随机写 I/O**

当插入一条新记录时，B+ 树需要定位到该 key 应该落入的叶子 page。如果 key 分布较随机（比如 UUID 做主键），每次插入大概率命中不同的 page。每个 page 的修改都意味着一次随机磁盘寻址。在 HDD 时代，一次随机寻址约 10ms，这意味着每秒只能做约 100 次随机插入——这是物理瓶颈。

**瓶颈二：页分裂的级联代价**

当一个叶子 page 写满后触发分裂（page split）：把一半数据搬到新 page，父节点增加一个路由项，如果父节点也满了就继续向上分裂。分裂过程涉及多次随机读写，且期间需要加锁，阻塞并发写入。

**瓶颈三：原地更新的 WAL 双写**

InnoDB 为了保证崩溃一致性，采用 WAL（Write-Ahead Log，即 redo log）。每次修改先写日志再改 page，这意味着每条写入实际产生两次磁盘写（日志 + 数据页）。虽然日志是顺序写很快，但 page 的最终刷盘仍是随机写。

**瓶颈四：缓冲池不命中时的放大**

当 Buffer Pool 无法容纳所有热 page 时（数据量远大于内存），每次写入大概率导致一次 page 读入（读磁盘）+ 一次 page 刷回（写磁盘），单条插入的实际 I/O 代价是 2 次随机操作。

### 9.3 对比总结

| 维度 | B+ 树 (MySQL InnoDB) | LSM-Tree (RocksDB) |
|------|---------------------|--------------------|
| 写入路径 | 找到 page → 原地修改 → 可能分裂 | 写 MemTable（内存） → 后台 flush/compact |
| 单次写入 I/O | 最坏 1 次随机读 + 1 次随机写 | 0（写内存即返回） |
| 写入吞吐 | 受限于随机 IOPS | 接近磁盘顺序写带宽 |
| 点查效率 | O(log N) 一次查到 | 可能需查多层 + Bloom Filter |
| 范围查询 | 叶子链表顺序扫描，天然高效 | 需多路归并各层数据 |
| 空间利用率 | 约 50-70%（分裂后页半满） | 接近 100%（Compaction 后紧凑） |
| 适合场景 | OLTP 读写均衡 | 写密集、日志、时序、大规模摄入 |

### 9.4 一个直觉性的对比

假设你每秒要插入 10 万条随机 key 的记录：

- B+ 树：需要 10 万次随机 page 定位和修改，即使有 Buffer Pool 缓冲，当数据量超过内存时也会退化为大量随机 I/O，吞吐骤降。
- LSM-Tree：10 万条全写入内存 MemTable（微秒级），攒够一批后一次性 flush 成顺序文件，后台异步 compact。前台写入吞吐基本等于内存写速度。

这就是为什么 RocksDB 在纯写入基准测试中能比 InnoDB 快 5-10 倍以上的根本原因。

## 十、RocksDB 中 LSM-Tree 的实际数据流——从 Put 到磁盘

以一次 `db->Put(key, value)` 调用为例，完整走通数据落盘的全过程：

**Step 1：写 WAL**

调用发生后，RocksDB 首先将这条写操作追加到 WAL 文件末尾（顺序写，极快）。这保证了即使进程崩溃，重启后可以从 WAL 恢复未落盘的数据。

**Step 2：写入 Active MemTable**

同时，数据被插入当前活跃的 MemTable（内存中的 SkipList）。到这里 `Put` 就返回了——用户感知到的写延迟极低。

**Step 3：MemTable 冻结 → Immutable MemTable**

当 Active MemTable 大小达到阈值（默认 64MB），它被冻结为 Immutable MemTable（只读），同时分配一个新的 Active MemTable 继续接收写入。

**Step 4：Flush 到 L0**

后台 Flush 线程将 Immutable MemTable 整体序列化为一个 SSTable 文件，写入 L0 层。这是一次纯顺序写：按 key 有序遍历 SkipList，依次写出 data block、index block、filter block、footer。

**Step 5：Compaction L0 → L1 → ... → Lmax**

后台 Compaction 线程监控各层文件数量和大小。当 L0 文件数超过阈值（默认 4 个），触发 L0 → L1 合并；当 L1 总大小超过阈值，触发 L1 → L2 合并，以此类推。合并过程是多路归并排序：读取待合并的多个 SSTable，按 key 排序输出为新的 SSTable，同时清理过期版本和 tombstone。

**数据流示意：**

```
Put(k,v)
  │
  ▼
┌─────────┐    ┌──────────────────┐    ┌─────────┐
│  WAL    │    │ Active MemTable  │    │ Immutable│
│(顺序写) │    │  (SkipList)      │───▶│ MemTable │
└─────────┘    └──────────────────┘    └────┬────┘
                                            │ Flush（顺序写）
                                            ▼
                    ┌───────────────────────────────────────┐
                    │  L0: SSTable files (key重叠)           │
                    └───────────────────┬───────────────────┘
                                        │ Compaction（顺序读+顺序写）
                                        ▼
                    ┌───────────────────────────────────────┐
                    │  L1: SSTable files (key不重叠, 10x大)  │
                    └───────────────────┬───────────────────┘
                                        │ Compaction
                                        ▼
                    ┌───────────────────────────────────────┐
                    │  L2: SSTable files (key不重叠, 100x大) │
                    └───────────────────────────────────────┘
```

## 十一、真实案例——Facebook 为什么用 MyRocks 替换 InnoDB

### 11.1 背景

Facebook 的 MySQL 集群存储了 PB 级的社交数据。他们发现 InnoDB（B+ 树）有两个严重问题：

1. **空间放大**：InnoDB 的 B+ 树页分裂后平均只有 50-70% 的填充率，加上 UNDO 日志和碎片，实际磁盘占用约为有效数据的 1.5-2 倍
2. **写放大**：InnoDB 的 double write buffer + redo log + data page flush，写放大约 10-30 倍

### 11.2 MyRocks 的效果

Facebook 用 RocksDB 作为 MySQL 的存储引擎（即 MyRocks），结果：

- **空间节省 50%**：SSTable 紧凑存储 + Zstd 压缩，有效数据比接近 1:1
- **写入吞吐提升 2-3 倍**：前台无随机 I/O
- **SSD 寿命延长**：总写入量减少（相比 InnoDB 的写放大更低）

当然代价是：点查延迟 P99 略有上升（需查多层），范围查询性能有所下降。但对于 Facebook 的写密集社交负载来说，这个 trade-off 完全值得。

### 11.3 为什么不是所有场景都替换

- **OLTP 读多写少场景**：B+ 树仍然更优，因为点查只需 1 次 I/O
- **强一致性事务场景**：InnoDB 的行级锁和 MVCC 实现更成熟
- **实时分析场景**：范围查询密集，B+ 树叶子链表扫描更高效

所以现实中的选择是：**根据 workload 特征选存储引擎**，没有绝对的优劣。

## 十二、LSM-Tree 在其他系统中的实际应用

### 12.1 TiKV / TiDB

TiDB 是分布式 NewSQL 数据库，其存储层 TiKV 底层使用 RocksDB。每个 TiKV 节点管理的数据以 Region 为单位（默认 96MB），每个 Region 的数据存储在本地 RocksDB 实例中。

为什么选 RocksDB 而不是直接用 B+ 树？因为 TiKV 的 Raft 日志复制本身就是顺序写密集的，再加上 MVCC 产生的大量版本数据需要高效写入和后台清理——这恰好是 LSM-Tree 最擅长的场景。

### 12.2 Cassandra

Cassandra 是 AP（可用性优先）型分布式数据库，天然是写优化设计。它的存储引擎就是 LSM-Tree：MemTable → CommitLog → SSTable → Compaction。配合其"写入不需要读"的设计（不需要先读旧值再修改），写入吞吐可以线性扩展。

### 12.3 ClickHouse 的 MergeTree

ClickHouse 用于 OLAP 分析，其核心存储引擎 MergeTree 虽然不是严格的 LSM-Tree，但借鉴了相同的思想：数据先写入内存小块，然后批量写入磁盘形成有序数据文件（parts），后台异步做 merge 合并。这使得 ClickHouse 可以在保持高速写入的同时提供快速的列式分析查询。

### 12.4 Kafka 的日志存储

Kafka 的消息存储虽然不是标准 LSM-Tree，但体现了相同的核心哲学：数据永远追加写入（append-only），不原地修改，通过 offset 顺序消费。这种"先接住再整理"的思想与 LSM 一脉相承。

## 十三、LSM-Tree 的优化方向——当前研究和工业界在做什么

### 13.1 降低写放大

写放大是 LSM-Tree 最大的工程痛点。当前主要优化方向：

**（1）WiscKey：Key-Value 分离**

核心思想：Compaction 时只对 key 做归并排序，value 存储在单独的 Value Log 文件中（类似日志追加）。由于 key 通常很小（几十字节），而 value 可能很大（几百字节到几 KB），分离后 Compaction 需要读写的数据量大幅减少。

代价：范围查询需要从 Value Log 中随机读取 value，适合 value 较大且点查为主的场景。

RocksDB 的 BlobDB 功能就实现了这个思想。

**（2）Tiered Compaction / Lazy Leveling**

在传统 Leveled Compaction 和 Size-Tiered Compaction 之间寻找混合策略。例如 RocksDB 的 Universal Compaction 允许同层堆积多个文件再一次性合并，减少合并次数。

**（3）SPDK + io_uring：绕过内核减少 I/O 开销**

使用用户态 I/O 接口直接操作 NVMe SSD，避免内核文件系统和 page cache 的额外开销，让 Compaction 的 I/O 吞吐接近硬件极限。

### 13.2 降低读放大

**（1）优化 Bloom Filter**

- MonkeyDB 论文：为不同层分配不同大小的 Bloom Filter（底层更大，因为底层被查到的概率更高）
- RocksDB 支持 Prefix Bloom Filter：对 key 前缀建 filter，适合前缀查询场景

**（2）Learned Index**

用机器学习模型（如分段线性函数）替代传统 B+ 树索引来定位 SSTable 内部的 key 位置，减少每次查找的比较次数和 I/O 次数。Google 在 Bigtable 中已经实验性部署了这种方案。

**（3）LSM-Tree 与缓存的联合优化**

通过精细化 Block Cache 管理（如区分 index block、data block、filter block 的缓存优先级），减少热 key 的磁盘访问次数。

### 13.3 降低空间放大

**（1）更激进的压缩**

对底层（存放大部分数据的层）使用更高压缩比的算法（如 Zstd），对上层使用更快的算法（如 LZ4）。RocksDB 支持按层配置不同压缩算法。

**（2）在线 GC 和 TTL**

为 key 设置过期时间（TTL），Compaction 时自动丢弃过期数据，避免过期数据持续占用空间。

### 13.4 降低尾延迟（P99/P999 延迟）

**（1）Rate Limiter**

RocksDB 内置 Rate Limiter，限制 Compaction 的 I/O 速率，避免 Compaction 大量占用磁盘带宽导致前台读写延迟飙升。

**（2）SubCompaction（并行 Compaction）**

将一次大的 Compaction 任务拆分为多个子任务并行执行，减少单次 Compaction 的持续时间，避免长时间的 Write Stall。

**（3）Direct I/O + 独立 Compaction 线程池**

将 Compaction 的 I/O 和前台 Read/Write 的 I/O 隔离到不同的线程池和 I/O 通道，减少相互干扰。

## 十四、LSM-Tree vs B+ 树——什么时候该选什么

### 14.1 选 B+ 树（MySQL InnoDB / PostgreSQL）的场景

- 读写比接近 1:1 或读多写少的 OLTP
- 需要高效范围查询（报表统计、分页查询）
- 对延迟敏感（需要稳定的 P99 延迟）
- 事务要求高（行级锁、MVCC 成熟）
- 数据量可控（能用内存覆盖热数据的索引）

### 14.2 选 LSM-Tree（RocksDB / Cassandra）的场景

- 写密集（写入量 >> 查询量）
- 时序数据、日志数据、监控数据
- 大规模数据摄入（数据量远超内存）
- 对空间效率要求高（需要压缩存储）
- 可接受略高的读延迟波动

### 14.3 混合方案

现代系统越来越多采用混合方案：

- **PolarDB / Aurora**：计算层用 B+ 树（兼容 MySQL），存储层用日志结构化设计
- **TiDB**：SQL 层兼容 MySQL（像 B+ 树一样查询），存储层用 RocksDB（LSM-Tree 写入）
- **CockroachDB**：同上，SQL 兼容 PostgreSQL，底层 Pebble（Go 版 LSM）

这种分层设计的哲学是：**上层提供 B+ 树的查询友好性，下层利用 LSM-Tree 的写入高效性。**

## 十五、这篇论文的核心思想——用 6 句话压缩

1. 高写入索引最大的敌人不是比较次数，而是**磁盘随机 I/O**。
2. B+ 树的"写入即到位"哲学在随机写密集场景下成本极高（随机 I/O + 写放大 + 页分裂）。
3. LSM-Tree 的核心思路是：**先在内存接住，再批量顺序落盘，通过后台归并逐步整理。**
4. 这种设计把 N 次随机写转化为 1 次顺序写，利用了随机 I/O 和顺序 I/O 之间 100-1000 倍的性能差距。
5. 代价是读路径更复杂（查多层）、有写放大（数据多次搬运）、空间放大（旧版本暂存）。
6. LSM-Tree 的本质不只是一个数据结构，而是一种系统设计哲学：**在写入吞吐和读取效率之间选择了写入，用"延迟整理"换"即时高速接纳"。**

## 十六、深度 Q&A

### Q1：为什么 RocksDB 用 SkipList 而不是红黑树做 MemTable？

三个原因：第一，SkipList 支持无锁并发读和细粒度并发写（通过 CAS），而红黑树旋转需要复杂的锁协调。第二，SkipList 的有序遍历非常自然（链表直接走），Flush 时直接顺序输出即可。第三，实测中 SkipList 的性能和红黑树相当甚至更好，且代码复杂度低得多。

### Q2：Bloom Filter 到底怎么帮助 LSM 读取？

每个 SSTable 文件自带一个 Bloom Filter，它是一个紧凑的位数组，可以在 O(1) 时间内告诉你"这个 key 一定不在这个文件里"（假阴率=0）或"这个 key 可能在这个文件里"（假阳率≈1%）。点查时先查 Bloom Filter，如果返回"不在"就直接跳过这个文件，省去一次磁盘读。对于一个 7 层的 LSM-Tree，Bloom Filter 平均可以省去 99% 的无效文件读取。

### Q3：什么是 Write Stall？如何避免？

Write Stall 发生在：L0 文件堆积过多（compaction 来不及消化），RocksDB 会主动降速甚至暂停前台写入，等 compaction 追上来。避免方式包括：增加 compaction 线程数、使用更快的存储介质、调大 L0 触发阈值、使用 Rate Limiter 平滑 compaction I/O。

### Q4：LSM-Tree 如何保证崩溃一致性？

通过 WAL（Write-Ahead Log）。任何写入在进 MemTable 之前都先追加到 WAL。如果进程崩溃，重启时从最后一个 checkpoint 开始回放 WAL，恢复 MemTable 中丢失的数据。已经 Flush 到 SSTable 的数据是安全的（SSTable 是原子写入的）。

### Q5：Compaction 期间能否继续读写？

可以。RocksDB 的 Compaction 是无锁的：正在被 compact 的旧 SSTable 文件在 compact 完成前不会被删除，读请求仍然可以访问它们。新的 SSTable 写完后，通过一次原子的 MANIFEST 更新切换版本，之后旧文件才被标记为可删除。写入不受 Compaction 影响（写的是 MemTable）。

### Q6：LSM-Tree 适合做主键索引吗？

适合，但需要权衡。RocksDB/MyRocks 中主键就是 LSM 的 key，点查需要查多层。对于写入量大于查询量的场景（如社交动态、IoT 时序）这是合理的。对于读写均衡的 OLTP，InnoDB 的聚簇 B+ 树索引仍然更优。

### Q7：LSM-Tree 的层数通常是多少？

取决于数据总量和放大因子。以 RocksDB 默认配置为例（放大因子 10，L1=256MB）：L1=256MB，L2=2.5GB，L3=25GB，L4=250GB，L5=2.5TB，L6=25TB。也就是说存储 1TB 数据大约需要 5-6 层。

## 十七、总结——作为一个系统工程师你应该带走什么

### 17.1 技术认知

读完 LSM-Tree，你应该建立的核心认知是：**数据结构的优劣从来不是绝对的，而是相对于底层硬件特性和业务负载模式而言的。** B+ 树在磁盘随机读友好的 OLTP 中是合理的；LSM-Tree 在写入密集、顺序 I/O 占优的场景中是合理的。

### 17.2 工程决策能力

当你面临存储引擎选型时，应该问自己：

1. 我的 workload 是写多还是读多？
2. 我的 key 分布是顺序的还是随机的？
3. 我能接受的读延迟 P99 是多少？
4. 我的数据量是否远超内存？
5. 我对空间利用率的要求是什么？
6. 我对 SSD 寿命是否敏感（写放大）？

### 17.3 系统设计思想

LSM-Tree 论文传达的最深层思想，远超过数据结构本身：

- **延迟整理（Deferred Processing）**：不必立刻做所有事情，先快速接住请求，后台慢慢整理
- **批量摊销（Batch Amortization）**：把 N 次昂贵的细粒度操作合并为 1 次廉价的粗粒度操作
- **分层分级（Tiered Architecture）**：利用存储层次（内存→SSD→HDD）各自的最优使用模式
- **以空间换时间，以后台换前台**：多占一些空间和后台 CPU，换来前台写入路径的极致简洁

这些思想不只在 LSM-Tree 中出现。你会在操作系统的 Page Cache、网络栈的 TCP 缓冲区、编译器的延迟求值、分布式系统的异步复制中反复看到它们。
